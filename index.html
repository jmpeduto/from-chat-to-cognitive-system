01 · Generative AI

What is an LLM: prediction, not thinking

A Large Language Model is a neural network with billions of parameters trained to predict the next token. It does not "think": it completes statistical patterns. The quality of the result depends on the quality of the context you provide.

02 · Generative AI

Models that exist today

Gemini for deep reasoning, Opus for clean code, Codex for testing, Copilot for autocomplete. The key is not which is "best" but which to use for each task.

03 · Generative AI

Chinese open source: frontier for pennies

Kimi K2.5, MiniMax M2.5 and GLM-5 offer ~90% of Opus/Gemini capacity at 1/20 the price. Frontier is no longer exclusive to the big players. The model router can use these engines for high-volume tasks.

04 · Generative AI

What an LLM is NOT

Not a database, does not learn from your conversations, does not understand like a human, is not deterministic. If you do not know this, you will have incorrect expectations and constant frustration.

05 · Context Window

The fixed-size whiteboard

The context is everything the model "sees": system prompt, history, files read, your instruction. The context window is the physical size of that whiteboard, measured in tokens.

06 · Context Window

Attention decay: more is not better

As you fill the window, the model does not fail: it degrades. Instruction drift, recency bias, generic code. It is subtle and dangerous because the output "looks" correct.

07 · Context Window

Compaction: forced amnesia

When the whiteboard fills up, the system generates a compressed summary and erases the original messages. You lose variable names, design decisions, user preferences.

08 · Context Window

How the industry tackles it

Long context (1M+ tokens), RAG, fine-tuning, memory layers, vector stores. Each approach has tradeoffs: cost, latency, precision, complexity. Nobody has solved it completely.

09 · Context Window

Why no solution is complete

Long context degrades quality. RAG depends on embeddings. Fine-tuning is slow. Memory layers are experimental. You need memory that survives sessions, compactions and model changes. That is where Engram comes in.

10 · Chat vs Agent

Chat answers text. Agent acts.

A chat generates text. An agent reads your code, edits files, runs tests and saves learnings. It is the difference between asking someone for advice and having a pair-programmer with access to your project.

11 · Chat vs Agent

Tools: the agent's capabilities

Terminal, file editor, search, memory. Each tool is an external capability. Without tools it is chat. With tools it is an agent. That simple.

12 · Context Evolution

It all started with AGENTS.md

The first idea: a giant markdown file with ALL the project rules. Style, patterns, anti-patterns, conventions, workflows. The agent read it at startup and "knew" how to work.

13 · Context Evolution

AGENTS.md grows infinitely

Each new convention bloated the file. In real projects it can reach thousands of lines. It consumes the context window before you write your first prompt.

14 · Context Evolution

First evolution: Skills

Instead of a giant file, a router (lightweight AGENTS.md) that points to specific SKILL.md files. Only the context you need for the current task is loaded. Dynamic context on demand.

15 · Context Evolution

Second evolution: sub-agents

Skills solves WHAT context to load. But a single agent keeps accumulating noise. The solution: delegate each phase to an ephemeral sub-agent with fresh context. Born, executes, reports and dies.

16 · Context Evolution

The current pattern: all together

Orchestrator (coordinates) + Skills (precise context) + Sub-agents (clean execution) + Persistent memory (continuity). The architecture that solves LLM limitations with engineering.

17 · God Agent

An agent that does everything: why it degrades

"Analyze, design, test and implement." A single agent fills its context to 80% before writing the first line. It compiles but ignores the decisions from step 1.

18 · God Agent

Divide and conquer: orchestrator + sub-agents

The orchestrator does not work: it coordinates. It delegates each task to a specialized sub-agent that receives only what it needs. Maximum precision, zero accumulated noise.

19 · God Agent

Fresh context: starting clean

Each sub-agent starts with a blank slate. It only receives the instructions for its phase. It does not carry noise from previous phases. That is the key architectural advantage.

20 · SDD Orchestrator

The philosophy: engineering applied to AI

Instead of "ask and pray", we structure work as an expert assembly line. Spec-Driven Development applies software engineering principles to the AI workflow.

21 · SDD Orchestrator

The specialized roles

Explorer (reads code), Proposer (defines what), Spec Writer (requirements), Designer (architecture), Task Planner (divides work), Implementer (writes code), Verifier (validates quality).

22 · SDD Orchestrator

The DAG: dependency flow

Proposal → [Specs || Design] → Tasks → Apply → Verify → Archive. Phases execute in order with checkpoints. Specs and Design can run in parallel.

23 · SDD Orchestrator

Real example: create users endpoint

/sdd:new → Explorer detects stack → Proposer defines scope → Spec + Design in parallel → Tasks splits into 4 → Implementer generates code → Verifier validates against specs.

24 · SDD Orchestrator

Why it scales

Context isolation, quality checkpoints, institutional memory. The Implementer is not distracted by the Explorer's doubts. If the Verifier rejects it, it does not reach the PR.

25 · SDD Orchestrator

SDD Agent Team: the library

Open source. Delegate-only orchestrator, sub-agents with skills, dependency DAG, Engram integration. Ready to use with any stack.

26 · Engram

The problem: compaction kills decisions

You spend 2 hours defining architecture. The system compacts. The agent no longer remembers you decided to use Clean Architecture with DI. It generates generic code.

27 · Engram

How the industry tackles it

RAG with vector stores, periodic fine-tuning, experimental memory layers, long context windows. Each has different cost, complexity and tradeoffs.

28 · Engram

My approach: signal memory, not total

We do not store everything (that is noise). We store signals: decisions, resolved errors, patterns. Structured observations with What/Why/Where/Learned.

29 · Engram

Architecture: Go + SQLite + FTS5

One binary. No external dependencies. SQLite with Full-Text Search 5. Connects via MCP to any agent: OpenCode, Claude Code, Cursor.

30 · Engram

The memory tools

mem_save (store decision), mem_search (search context), mem_context (retrieve previous session), mem_session_summary (summary for the future). 13 tools, 4 critical.

31 · Engram

The Engram Loop: two sessions

Session 1: design → mem_save. Session 2: implement → mem_search retrieves decision → consistent code. Without Engram session 2 ignores session 1. With Engram there is real continuity.

32 · Engram

Engram vs alternatives: why lightweight wins

No server, no embeddings, no cloud infrastructure. A local binary that works offline. For teams starting out, simplicity is the most important feature.

33 · Skills Registry

The problem of the giant prompt

If you put all the rules in the system prompt, you saturate the window before you start. The agent ignores rules and generates generic code. We already saw why in the evolution module.

34 · Skills Registry

AGENTS.md router + SKILL.md module

AGENTS.md as a lightweight index that maps intentions to skills. Each SKILL.md has patterns, anti-patterns and templates. Only what you need is loaded.

35 · Skills Registry

skill-creator: the team teaches the agent

The institutional knowledge belongs to the team. With skill-creator each person creates their own skills: conventions, patterns, checklists. The agent learns from the real team.

36 · Stack Completo

Ecosystem: Engram + SDD + Skills

Engram gives continuity, SDD gives structure, Skills gives consistency. Together they solve the three LLM problems: amnesia, degradation and genericness.

37 · Stack Completo

Model router: one engine per phase

There is no "best model". Gemini for architecture, Opus for implementation, Codex for testing. The orchestrator rotates engines according to the DAG phase.

38 · Stack Completo

Result: speed + consistency + quality

Speed through automation. Consistency through skills. Quality through verification against specs. The developer shifts from writing code to directing and validating code.

39 · Closing

The difference: architecture around the model

The model is the engine. But an engine without a chassis, without steering and without brakes is not a car: it is a hazard. The architecture around the LLM defines whether you have a toy or a tool.
